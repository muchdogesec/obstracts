## Overview

Organisations subscribe to lots of blogs for security information. These blogs contain interesting indicators of malicious activity (e.g. malicious URL).

To help automate the extraction of this information, Obstracts automatically downloads blog articles and extracts indicators for viewing to a user.

It works at a high level like so:

1. A feed is added to Obstracts by user
2. Obstracts uses history4feed as a microservice to handle the download and storage of posts.
3. The HTML from history4feed is converted to markdown.
4. The markdown is run through txt2stix where txt2stix pattern extractions are run (these are defined by staff).
5. A set of STIX bundles are generated for each post of the blog
6. The bundles are stored in ArangoDB using stix2arango
7. A user can access the bundle data or specific objects in the bundle via the API
8. As new posts are added to remote blogs, user makes request to update blog and these are requested by history4feed

### Feeds

Feeds are essentially subscriptions to blogs. Blogs contain items (aka posts). Each post has a text description (the content) that might contain security information to be extracted.

In Obstracts a user is limited to the number of feeds they can view (via subscribing to it).

Only staff users can add new feeds for ingestion.

### A short note on file2txt

history4feed delivers a HTML file.

One of the benefits of using Obstracts for a user is that 

1. the blog is converted to a more developer friendly format (markdown)
2. the text in the images embedded in the blog posts are turned into text so that txt2stix can extract data from them.

To use with Obstracts the content first needs to be converted to Markdown to allow for

* extraction of content using txt2stix
* to render the content without styling issue (because of messy HTML) in the UI.

file2txt can be found here:

https://github.com/signalscorps/file2txt

file2txt is run with the following settings;

```shell
python3 file2txt.py \
    --mode html \
    --input_filepath '<FILE WITH POST CONTENT FROM FTF API>' \
    --output_filename '<DOC>' \
    --defang true
```

This will product an file (`output_filename`) with the blog post content (inc. images in post converted to text).

### A short note on txt2stix

A lot of the functionality of Obstracts uses the txt2stix library under the hood to work with the markdown generated by file2txt. txt2stix can be found here;

https://github.com/signalscorps/txt2stix

The staff can decide what Extractions and Whitelists from this library are used.

Thus as the txt2stix code is updated (and pulled onto server), the options avaiable to staff should automatically reflect any changes (e.g. if new Extraction is added to the main txt2stix repository, staff should see this in the staff UI).

The command run for txt2stix is as follows;

```shell
python3 txt2stix.py \
    --relationship_mode '<MODE SET PER PLAN>' \
    --input_file '<BLOG MARKDOWN FILE AS CONVERTED BY FILE2TXT>' \
    --name '<BLOG TITLE FROM FTF>' \
    --labels '<CATEGORIES FROM FTF>' \
    --tlp_level clear \
    --confidence 0 \
    --use_extractions '<EXTRACTIONS SET PER PLAN>' \
    --use_aliases '<SET BY STAFF IN STAFF INTERFACE>' \
    --use_whitelists '<SET BY STAFF IN STAFF INTERFACE>'
```

### Extractions

Extractions are performed by txt2stix. txt2stix identifies potential strings in text (a blog post in Obstracts) related to security content (e.g. IPv4 addresses). txt2stix has a list of extractions it can perform. 

If an extraction finds a value in a blog post text, it is extracted, and these extractions are converted to STIX objects.

Staff user defines if a txt2stix extraction is enabled / disabled in the staff interface. There is a page in the app which shows users the enabled extractions. Enabled extractions are applied to all posts being processed.

txt2stix also creates relationships between extracted objects, either standard or AI mode. In

* standard mode, basic relationships exist.
* AI mode, smarter relationships between objects based on descriptions in the original text are made.

In Obstracts, the ability to view AI relationships is based on plan. Regardless, both relationship modes are run on all imported blogs as standard to create a bundle. The user just sees the version they have chosen.

Similarly there is both a basic and advance extraction mode.

* Basic mode includes only pattern extractions: `pattern_ipv4_address_only` , `pattern_ipv4_address_cidr` , `pattern_ipv4_address_port` , `pattern_ipv6_address_only` , `pattern_ipv6_address_cidr` , `pattern_ipv6_address_port` , `pattern_domain_name_only` , `pattern_domain_name_subdomain` , `pattern_url` , `pattern_url_file` , `pattern_url_path` , `pattern_host_name` , `pattern_host_name_subdomain` , `pattern_host_name_url` , `pattern_host_name_file` , `pattern_host_name_path` , `pattern_file_name` , `pattern_directory_windows` , `pattern_directory_windows_with_file` , `pattern_directory_unix` , `pattern_directory_unix_file` , `pattern_file_hash_md5` , `pattern_file_hash_sha_1` , `pattern_file_hash_sha_256` , `pattern_file_hash_sha_512` , `pattern_email_address`
* Advanced mode includes all basis extractions and: `pattern_mac_address` , `pattern_windows_registry_key` , `pattern_user_agent` , `pattern_autonomous_system_number` , `pattern_cryptocurrency_btc_wallet` , `pattern_cve_id` , `pattern_cpe_uri` , `pattern_bank_card_mastercard` , `pattern_bank_card_visa` , `pattern_bank_card_amex` , `pattern_bank_card_union_pay` , `pattern_bank_card_diners` , `pattern_bank_card_jcb` , `pattern_bank_card_discover` , `pattern_iban_number` , `pattern_phone_number`

In Obstracts, the ability view extractions is based on plan. Regardless, both extraction modes are run on all imported blog posts as standard to create a bundle. The user just sees the version they have chosen.

In short, this means for every blog imported the following bundles are created:

* standard relationship mode + basic extractions bundle
* standard relationship mode + advanced extractions bundle
* ai relationship mode + basic extractions bundle
* ai relationship mode + advanced extractions bundle

### A short note on stix2arango

All extractions and STIX objects are stored in ArangoDB. stix2arango already has the logic to handle this.

https://github.com/signalscorps/stix2arango

A skeleton stix2arango command to insert the bundles generated for each blog could be run as follows;

```shell
python3 stix2arango.py \
    --file '<BUNDLE GENERATED BY TXT2STIX>' \
    --database Obstracts_production \
    --collection '<BLOG ID>' \
    --ignore_embedded_relationships false
```

Where `<BLOG ID>` is the Obstracts ID for the blog.

### Whitelists

txt2stix also has a concept of Whitelists. Whitelists contain values that might match a security object (e.g. domain google.com) that are not malicious and therefore should not be extracted from the report.

Staff user defines if a txt2stix whitelist is enabled / disabled in the staff interface. There is a page in the app which shows users the enabled whitelists. Enabled whitelists are applied to all posts being processed.

### Aliases

txt2stix also has a concept of Aliases. Aliases modify input text to change values.

Staff user defines if a txt2stix aliases is enabled / disabled in the staff interface. There is a page in the app which shows users the enabled aliases. Enabled aliases are applied to all posts being processed.

## Workflows

### Staff user adding blogs

Currently, for stability (to ensure blogs ingest correctly) only staff user can add new feeds.

A staff user does this in the staff interface by specifying:

* fulltextfeed feed URL (url)
* earliest post (date)

Once submitted the following happens. 

* Obstract sends URL to FullTextFeed API (inc backfill date)
  * it is assumed the URL has already been added to fulltext feed. If this is not the case, staff user must add it on fulltext feed first, else will get a does not exist error.
* Obstracts downloads all blog posts from FullTextFeed from now until the earliest post specified
* Each blog article content (description) is converted from HTML to markdown using file2txt. Defang is always set to true
* The markdown is then run through txt2stix. All enabled extractions and whitelists set in staff interface are applied.
* The output objects are stored in ArangoDB using stix2arango in a database called `obstracts_database` with a collection for each blog (both edge and vertex collection for each blog)
* The blogs and posts exposed via API (with a layer of authentication on top to manage user access)

As the backfill is happening, feed is in state unpublished, and is not visible to any users.

On completion of backfill staff user can check backfill quality and publish feed.

It will then be available for all users to subscribe too via the feed library. The feed library shows all the staff 

## Checking blogs for new posts

Every 6 hours, Obstracts will query the FullTextFeed API for new posts for each blog.

If new posts are detected, they are imported as described above (starting at file2txt).

Note, some blog posts (that might have already been imported) are updated over time, put another way; a previously ingested blog might have been modified. Where a blog with the same `<link>` as one that already exists in Obstracts, first the old record and bundles are deleted, and then the reingest happens.

## Authentication

All authentication to UI is done via Auth0.

We use Auth0 because in future we will add more Signals Corps products. The idea is user creates a Signals Corps account once, and it provides them access to all products.

The authentication should be designed so that a user can have one login for future Signal Corps products. The authentication should be aware of the plan a user is on for each product.

As search, when new user created, it is given Auth0 scope for the current Obstracts plan they are on. When user registers for a new account, they are automatically assigned to the default plan.

A user should validate their email address (using native Auth0 verification) before the can log into their account. Thus we need to check Auth0 email verification status after sign up.

Auth0 docs: https://auth0.com/docs/quickstart/webapp/python/interactive

##### POST Create a Feed

```shell
POST <HOST>/api/v1/feeds/
```

Body:

```json
{
  "url": "string"
}
```

Use this endpoint to create to a new feed.

The url value used should be a valid RSS or ATOM feed URL. For example

* good = https://www.bellingcat.com/feed/
* bad = https://www.bellingcat.com/

The server will validate that the document url resolved to is valid ATOM/RSS. If it is not valid, the Feed will not be created and an error returned.

If a feed `url` already exists in the database, that record will be returned (vs. adding a new feed).

```shell

```

```json
{
    "feeds": [
        {
            "id": "<FEED ID>",
            "datetime_added": "<DATETIME ADDED>",
            "job_id": "<JOB ID RELATED TO FETCHING POSTS>",
            "state": "PENDING"
        }
    ]
}
```

Note, items in the remote feedd are added over time, a feed can be identified by a unique `link` and `pubDate`.

If the item has a unique `link` and `pubDate` (not found in the database) it is considered new, and should be added.

It is possible items in a feed are updated (and thus reappear in a later feed update request), in such cases a `link` and `pubDate` for a item in a feed will already match one already indexed in the database. If this happens, the latest copy of the post replaces the previous content for that item.

    * value field = `key`
* `autonomous-system`
    * value field = `number`
* `user-agent`
    * value field = `string`
* `cryptocurrency-wallet`
    * value field = `address`
* `cryptocurrency-transaction`
    * value field = `hash`
    * value field = `symbol`
* `bank-card`
    * value field = `number`
* `bank-account`
    * value field = `iban_number`
* `phone-number`
    * value field = `number`

Note the value field mapping is designed to show the field user will be most interested in (and thus what we expose for a user to search). e.g. for IP address, a user most wants the actual IP address (e.g. 1.1.1.1) whichy is held in the `value` property of the `ipv4-addr` object.

Accepts URL parameters;

* `types` (dictionary of STIX object types, shown above): allows results to be filtered on `data.type`
    * default: none
    * required: false
* `value` (string): searches through the value fields shown mapped above. Is wildcard search.
    * default: none
    * required: false
* `post_id`: allows results to be filtered using a post id
    * default: none
    * required: false
* `page`
    * default is 0
* `sort`:
    * type_descending (default)
    * type_ascending
